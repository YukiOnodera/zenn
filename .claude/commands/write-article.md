## Phase 0: ヒアリング

まず、以下をユーザーに質問してください。

「今日学んだことを教えてください。トピック名だけでなく、気づいたこと・詰まったこと・試したこと・参考にしたリソースなど、メモ書きレベルで構いません。」

ユーザーの回答を受け取ったら、内容を読んで**記事化するうえで確認したいことがあれば質問**してください。質問がなければその旨を伝えてPhase 1に進む。

## Phase 1: スタイル研究

以下の既存記事を3本すべて読み、筆者の文体・構成・口調を把握する。

- articles/1password-k8s-integrations.md
- articles/aws-ecr-cost-debug.md
- articles/chezmoi-use-template.md

特に注目すること：
- 書き出しの言い回し（「今回は〜について調査します」等）
- 見出し構成のパターン（## はじめに → 主要セクション → ## おわりに）
- 技術用語の初出時の説明スタイル
- 文末表現・全体の口調

## Phase 2: リサーチ

ヒアリング内容を踏まえ、不足している情報をWeb検索で補う：
- 概要・目的・背景
- 主要な概念・仕組み
- 具体的なユースケースや設定例
- よくある落とし穴・注意点
- 公式ドキュメントのURL

## Phase 3: アウトライン作成

Phase 1・2 を踏まえ、記事の構成を決めてユーザーに提示する。承認を得てから執筆に進む。

## Phase 4: 執筆

以下のルールで `articles/[slug].md` に記事を書く。スラッグは英数字とハイフンのみ（例: aws-vpc-endpoints）。

### フロントマター（必須）
```yaml
---
title: [日本語タイトル（20〜40文字）]
emoji: [内容に関連する絵文字]
type: tech
topics:
  - [英語タグ1]
  - [英語タグ2]
published: false
---
```

### 執筆ルール
- 書き出し：「今回は〜について[調査/まとめ/紹介]します」
- `##` で大セクション、`###` で小項目
- 重要な技術用語は初出時に **ボールド**
- コードブロックには必ず言語指定（```sh, ```yaml 等）、ファイル名がある場合は ` ```sh:filename ` 形式
- 外部リンクは `[説明テキスト | サイト名](URL)` 形式
- 補足情報は `:::message` ブロックを活用する
- 手順の補足や折りたたみたい情報は `:::details タイトル` を活用する
- 文末は「です・ます」調
- 1,500〜2,500字程度
- 最後に `## おわりに` で2〜3文の締め

## Phase 5: レビュー

執筆後、以下の観点でセルフレビューを行い、問題があればその場で修正する。

### ファクトチェック
- ヒアリング内容と記事の内容が一致しているか
- リサーチで追加した情報は公式ドキュメントに裏付けがあるか
- 裏付けが取れない情報は削除するか、「〜とのことです」などと明示する
- コマンド・設定値・APIの仕様に誤りがないか

### 読みやすさ
- 一文が長くなりすぎていないか（目安：60字以内）
- 専門用語を説明なしに使っていないか
- 図・コードブロック・messageブロックを適切に使って視覚的に整理されているか
- 見出しだけ読んで記事の流れが把握できるか

レビューと修正が完了したら、修正内容の概要をユーザーに報告してPhase 6に進む。

## Phase 6: 公開準備・Push

ユーザーから公開の許可が出たら、以下を実行する。

1. フロントマターの `published_at` を**翌日の19:00（JST）** に設定する
   ```yaml
   published: true
   published_at: YYYY-MM-DD 19:00 +0900
   ```
2. 以下のコマンドでコミット・Pushする
   ```sh
   git add articles/[slug].md
   git commit -m "add: [記事タイトル]"
   git push origin main
   ```
